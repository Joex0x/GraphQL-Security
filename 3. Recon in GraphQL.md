# Fingerprinting GraphQL

- *Server fingerprinting* is the operation of identifying information about the target’s running services and their versions.
- For example, send HTTP HEAD request using cURL.
- GraphQL fingerprinting relies on the observation of various discrepancies between implementations of GraphQL servers. Here are a few examples:
    - Inconsistencies in error messages
    - Inconsistencies in response outputs to malformed GraphQL queries
    - Inconsistencies in response outputs to properly structured queries
    - Inconsistencies in response outputs to queries deviating from the GraphQL specification
- For example, we send malformed query to implementation like Sangria, a Scala-based GraphQL server:
    
    ```graphql
    queryy {
    	__typename
    }
    ```
    
    - It responds with :
        
        ```graphql
        {
        	"syntaxError": "Syntax error while parsing GraphQL query.
        	Invalid input \"queryy\", expected ExecutableDefinition or
        	TypeSystemDefinition (line 1, column 1):\nqueryy {\n^",
        		"locations": [
        		{
        			"line": 1,
        			"column": 1
        		}
        	]
        }
        ```
        
- While if we send the same query to implementation like HyperGraphQL-a Java-based GraphQL server- it respond with :
    
    ```graphql
    {
    	"extensions": {},
    	"errors": [
    		{
    		"message": "Validation error of type InvalidSyntax: Invalid query syntax.",
    		"locations": [
    			{
    				"line": 0,
    				"column": 0,
    				"sourceName": null
    			}
    		],
    		"description": "Invalid query syntax.",
    		"validationErrorType": "InvalidSyntax",
    		"queryPath": null,
    		"errorType": "ValidationError",
    		"extensions": null,
    		"path": null
    		}
    	]
    }
    ```
    
- While if we send the same query to implementation like Graphene -a Python GraphQL server- it respond with :
    
    ```graphql
    {
      "errors": [
        {
          "message": "Syntax Error GraphQL (1:1) Unexpected Name \"queryy\"\n\n1: queryy {\n   ^\n2: \t__typename\n",
          "locations": [
            {
              "line": 1,
              "column": 1
            }
          ]
        ]
     }
    ```
    
- The 3 responses are different.
    - Use https://github.com/Escape-Technologies/goctopus

## Detecting Servers with Graphw00f :

- Graphw00f is currently the only tool available for GraphQL server fingerprinting.
- It can detect many of the popular GraphQL server implementations and provide meaningful information.
- You can use it with this command :
    
    ```bash
    cd ~/graphw00f
    python3 main.py -f -t http://localhost:5013/graphql
    ```
    
    - `-f` ⇒ To enable fingerprint.
    - `-t` ⇒ To specify target.
- It provide us with *attack surface matrix* is essentially knowledge about the security posture of the GraphQL implementation.

## Analyzing Results :

- Examples from the consideration in *attack surface matrix :*
    - Field Suggestions ⇒ inform client about spelling mistakes and suggests alternative options.
    - Query Depth Limit ⇒ prevent DoS attacks that may abuse conditions such as cyclical node relationships in schemas.
    - Query Cost Analysis⇒ prevent DoS attacks that stem from computationally complex queries.
    - Automatic Persisted Queries ⇒ is a caching mechanism. It allows the client to pass a hash representing a query as a way to save bandwidth and can be used as a security control with an allow list of safe queries.
    - Debug Mode ⇒ provide additional information in the response for debugging purposes.
    - Batch Requests ⇒ is a feature that provides clients with the ability to send a sequence of queries in a single HTTP request. Batch queries are a great vector for DoS attacks.

# Querying GraphQL by Using Introspection

## The __typename Field :

- To extract fields of query, you can use *Meta-fields.*
- *Meta-fields* are built-in fields that GraphQL expose to clients like `__schema` and `__typename` .
- `__typename` returns the name of the object type being queried like :
    
    ```graphql
    query {
    	pastes {
    		__typename
    	}
    }
    ```
    
- You can use `{ __typename }` to get the root query object and if the endpoint is not GraphQL it responds with error, so you can use it with Nmap to check the endpoint (GET based):

## Introspection:

- Introspection provides information about the various types and fields the GraphQL schema supports.
- It is enable by default in all GraphQL server implementations, but may be configured to disabled
- It uses *meta fields* like `__schema` , `__type` to return the schema data, for example :
    
    ```graphql
    query {
      __schema {
        types {
          name
        }
      }
    }
    
    ```
    
- You can can drive deeper with this query :
    
    ```graphql
    query IntrospectionQuery {
      __schema {
        queryType { name }
        mutationType { name }
        subscriptionType { name }
        types {
          kind
          name
          fields {
            name
            args {
              name
            }
          }
        }
      }
    }
    
    ```
    
    - `queryType`, `mutationType`, `subscriptionType` ⇒ Shows the *entry points* for data access.
    - `types` ⇒ Shows the **types**, whether they are object, scalar, enum, etc.
    - `fields` ⇒ The fields available inside each type.
    - `args` ⇒ What arguments (input) are required for those fields.
- For a comprehensive query use this : https://gist.githubusercontent.com/craigbeck/b90915d49fda19d5b2b17ead14dcd6da/raw/e50819812a7a8a95b303ac0ea1464e2679e3e4bc/introspection-query.graphql
