- **Authentication** is the mechanism by which a client proves their identity to a server.
- **Authorization** controls are responsible for granting access to data.

# The State of Authentication and Authorization in GraphQL

## In-Band vs. Out-of-Band :

- **In-Band** is auth logic is *inside* the GraphQL API server itself.
- **Out-of-Band is au**th logic is *outside* the GraphQL API delegated to other components.
- Auth layers :
    1. **Gateway layer** ⇒ Handles **authentication** (verifying who the user is).
    2. **API layer** ⇒ Contains the GraphQL endpoint.
    3. **Persistence layer** ⇒ Database or datastore.

## Common Approaches :

### HTTP Basic Authentication :-

- Older version: `admin:pass123@example.com`.
- It involves the inclusion of a Base64-encoded username and password in the request header.
- The header looks as follows: `Authorization: Basic <base64_encoded_credential>` .
    - https://www.base64encode.org/

### OAuth 2.0 and JSON Web Token :-

- OAuth 2.0 is a **framework** that lets a client (e.g., a web app) get **temporary, limited access** to a protected resource (e.g., a GraphQL API) *on behalf of a user*.
- How it works :
    1. Click “Log in with Google” and the app will redirect you to it.
    2. Google ask you if you want to give app access.
    3. If you accept, Google sends the app an **authorization code**.
    4. The app exchanges that code for an **access token**.
    5. The app sends the **access token** with each request.
- JWT is **one way to represent the access token**. It’s a Base64-encoded string with 3 parts: `<header>.<payload>.<signature>` , For example:                        https://www.jwt.io/
    
    ```
    eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9.eyJ0eXBlIjoiYWNjZXNzIiwiaWF0Ijo
    xNjU2NDY0MDIyLCJuYmYiOjE2NTY0NjQwMjIsImp0aSI6ImY0OThmZmQxLWU0YzctNGU
    5Mi05ZTRhLWJiNzRiZmVjZTE4ZiIsImlkZW50aXR5Ijoib3BlcmF0b3IiLCJleHAiOjE
    2NTY0NzEyMjJ9.NHs6JiLDONJsC9LpJzdBB8enXzIrqI0Cvqojj8SqA4s
    ```
    
- Header in base64-decoded :
    
    ```json
    {
      "typ": "JWT",
      "alg": "HS256"
    }
    ```
    
    - **typ** ⇒ token type (JWT)
    - **alg** ⇒ signing algorithm (e.g., HS256 for HMAC with SHA-256)
    
    ⚠ **Attack risk:** If a server accepts `alg: none` and doesn’t check properly, an attacker can send an unsigned JWT to bypass authentication.
    
- Payload in base64-decoded :
    
    ```json
    {
      "type": "access",
      "iat": 1656464022,
      "nbf": 1656464022,
      "jti": "f498ffd1-e4c7-4e92-9e4a-bb74bfece18f",
      "identity": "operator",
      "exp": 1656471222
    }
    ```
    
    - **type** ⇒ purpose (e.g., "access" token)
    - **iat** ⇒ issued at time
    - **nbf** ⇒ not before (start time of validity)
    - **exp** ⇒ expiration time
    - **identity** ⇒ user identifier (could be username, user ID, etc.)
- **Signature** is Created by hashing `header + payload` with a secret key and used by the server to confirm the token hasn’t been altered.

### GraphQL Modules :-

- It’s a JavaScript/TypeScript library separates GraphQL schemas into smaller, reusable modules contains (**Schema definitions,** **Resolvers, Middleware logic).**

```graphql
extend type Query {
  me: User
}

type Mutation {
  login(username: String!, password: String!): User
  signup(username: String!, password: String!): User
}

extend type User {
  username: String!
}
```

- **Query**:
    - `me` ⇒ returns the currently authenticated user's info (type `User`).
    - The resolver typically reads the authenticated user from:
        - JWT in the Authorization header
        - Session cookie
        - Request context (`context.user`)
    - If no user is authenticated, it often returns `null` or errors.
- **Mutations**:
    - `login(username, password)` ⇒ logs the user in, returns a `User` object.
    - `signup(username, password)` ⇒ creates a new account, returns a `User` object.
- **Extend User type**:
    - Adds a `username` field to the `User` type.

### Normal GraphQL auth ⇒

- Login is REST-based
- One credential attempt per request
- Rate limits are enforced at HTTP layer

❌ No batching

❌ No alias abuse

### Without GraphQL Modules (normal GraphQL) ⇒

You often have to:

- Guess if login exists
- Test REST endpoints
- Infer auth from behavior

❌ Slower recon

❌ Hidden entry points

📌 **As a pentester, GraphQL Modules saves you time.**

### GraphQL Shield :-

- it is another middleware library for generating an authorization layer in GraphQL APIs by defining rules that either permit or deny client access.

```graphql
const permissions = shield({
  Query: {
    frontPage: not(isAuthenticated),
    fruits: and(isAuthenticated, or(isAdmin, isEditor)),
    customers: and(isAuthenticated, isAdmin),
  },
  Mutation: {
    addFruitToBasket: isAuthenticated,
  },
  Fruit: isAuthenticated,
  Customer: isAdmin,
})
```

- **Query.customers :** `and(isAuthenticated, isAdmin)` ⇒ Must be logged in **AND** **admin**.
- **Mutation.addFruitToBasket :** `isAuthenticated` ⇒ Any logged-in user can run this mutation.
- **Fruit type :** `isAuthenticated` ⇒ Can only see `Fruit` fields if logged in.

### Your checklist:

- Are **all queries listed**?
- Are **all mutations listed**?
- Are **new schema additions protected**?
- Are **extended types covered**?

### Schema Directives :-

- **Schema directives** `@` are annotations in the GraphQL schema that change how certain parts of the schema behave.

```graphql
type Query {
  me: User @auth(requires: "USER")
  adminPanel: AdminDashboard @auth(requires: "ADMIN")
}

type User {
  id: ID!
  username: String!
}
```

- The `@auth` directive enforces role-based access.
- Typical flow:
    1. Client sends a request with a JWT
    2. JWT contains roles / groups / scopes
    3. Directive logic:
        - Extracts claims from `context.user`
        - Compares them with directive arguments
    4. Access is allowed or denied **before resolver execution**
- `requires: "USER"` means the client must have the `USER` role in their **JWT payload** or session context.

| Directive Name | Argument Name | Argument Type |
| --- | --- | --- |
| `@auth` | requires | String |
| `@protect` | role | String |
| `@hasRole` | role | String |

### IP-Based Allow Listing :-

- Some APIs does not enforce authentication, but depend on IP that sent in header like : `X-Forwarded-For`, `X-Real-IP`, `X-Originating-IP`, and `X-Host`.
- For example :
    
    ```bash
    curl -X POST http://localhost:5013/graphql -d '{"query":"{ __typename }"}'
    -H "Content-Type: application/json" -H "X-Forwarded-For: 10.0.0.1"
    ```

# Authentication Testing

## Detecting the Authentication Layer :

- One of the best ways to determine whether a target GraphQL application is protected by an authentication layer is by sending it a canary query.

### HTTP Status Codes :-

- You may receive a *403 Fobidden Error* and that means out-of-band authentication and authorization control.

### Error Messages :-

- Error messages can  reveal the presence of authentication controls and the type of authentication the API requires.
- Common GraphQL Authentication Errors :
    
    
    | **Error message** | **Possible authentication implementation** |
    | --- | --- |
    | Authentication credentials are missing. Authorization header is required and must contain a value. | OAuth 2.0 Bearer with JSON Web Token |
    | Not Authorised! | GraphQL Shield |
    | Not logged in Auth required
    API key is required | GraphQL Modules |
    | Invalid token! Invalid role! | graphql-directive-auth |

### Authentication-Related Fields :-

- Another great way to detect an authentication layer is to use **introspection queries** to identify any authentication-related query or mutation operations.

## Using Allow-Listed Operation Names :

- Some of these deployments use *operation name based allow lists* to reject all unauthenticated requests unless their operation names (defined by client) are in an allow list.
- If this operation name is allowed :
    
    ```graphql
    mutation RegisterAccount {
      register(username: "operator", password: "password") {
        user_id
      }
    }
    ```
    
    Attack can exploit it using : 
    
    ```graphql
    mutation RegisterAccount {
      withdrawal(amount: 100.00, from: "ACT001", dest: "ACT002") {
        confirmationCode
      }
    }
    ```

# Authorization Testing

## Detecting the Authorization Layer :

### Finding Schema Directives :-

- Use this query:
    
    ```graphql
    {
      __schema {
        directives {
          name
          args {
            name
          }
        }
      }
    }
    ```
    
- Running introspection query will return a list of all the query and schema level directives in the target server.
- If you notice the `@auth`, `@authorize`, `@authorization` and `@authz` directive in the list, you can assume that the schema supports it.

### Finding Authentication Directives in the Schema :-

- It may applied by authentication directives like :
    
    ```graphql
    Type User {
    	id: ID
    	username: String
    	email: String
    	password: String @auth(requires: ADMIN)
    	role: String
    }
    ```
    

### Enumerating Paths with graphql-path-enum :-

- You should determine all possible paths to a given object type like :
    
    ```graphql
    type Query {
    pastes(public: Boolean, limit: Int, filter: String): [**PasteObject**]
    paste(id: Int, title: String): **PasteObject**
    --snip--
    users(id: Int): [UserObject]
    readAndBurn(id: Int): **PasteObject**
    search(keyword: String): [SearchResult]
    audits: [AuditObject]
    deleteAllPastes: Boolean
    me(token: String): UserObject
    }
    ```
    
- You can use this command :
    
    ```bash
    └─$ ./graphql-path-enum -i introspection.json -t PasteObject
    Found 3 ways to reach the "PasteObject" node:
    - Query (pastes) -> PasteObject
    - Query (paste) -> PasteObject
    - Query (readAndBurn) -> PasteObject
    ```
