# Request Forgery And Hijacking
- *Request forgery* occurs when an attacker is able to carry out an action on behalf of a client or server.
- By contrast, *hijacking* refers to the ability to steal another user’s session.

# Cross-Site Request Forgery (CSRF)

- CSRF is a client-side attack that causes victims to execute unwanted actions on a website to which they are authenticated.
- How it works :
    1. The attacker writes code and embeds it in a website that they operate.
    2. They then force the victim to visit that site.
    3. When the code executes in the victim’s browser, it forges and sends a request to the server.
- The browser sends necessary information in every HTTP request it makes to the site, such as session cookies and web servers have no way to distinguish if the request is legitimate or not.

## Locating State-Changing Actions :

- To find impactful CSRF vulnerabilities, try extracting the list of available mutations and searching for ones that give you a foothold in the application or allow you to escalate your existing privileges.
- Execute this introspection query to fetch mutations :
    
    ```graphql
    query {
      __schema {
        mutationType {
          fields {
            name
          }
        }
      }
    }
    ```

## Preventing CSRF :

### The SameSite Flag :-

- Browsers have started supporting a special HTTP cookie attribute called *SameSite* to allow developers to decide whether the client browser should attach the cookie when making cross-site requests.
- The SameSite cookie attribute accepts three values:
    - Strict ⇒ Send the cookie only when the user is browsing within the same origin
    - Lax ⇒ Send cookies only when the request uses HTTP GET and was not initiated by a script, such as by top-level navigation.
    - None ⇒ Send the cookie on cross-site requests, effectively providing no protection.
- The default value is set to Lax.

### Anti-CSRF Tokens :-

- The server expects the client to pass an anti-CSRF token on every request. When a server sees an incoming request without such a token, the server rejects that request.
- Here are a few ways an attacker might go about bypassing anti-CSRF tokens:
    - Removing the CSRF token value.
    - Removing the CSRF parameter and token value altogether.
    - Reusing a CSRF token in subsequent requests.
    - Replacing the CSRF token with a random string of the same character length.
    - Brute-forcing the CSRF token.

# Server-Side Request Forgery (SSRF)

- It allows an attacker to **perform requests on behalf of an impacted server.**
- Using It, attackers could force the server to do :
    - Establish connections to **internal services.**
    - Provide access to restricted **network zones, internal APIs, and sensitive data**
- How it happens :
    1. Attacker provides a URL to site.
    2. Site fetch the data from URL and save it anywhere.
- An attacker could also supply a variety of URLs as input, including URLs that contain addresses of private and non-routable IP addresses.

## Understanding the Types of SSRF :

- *Blind* SSRF vulnerabilities do not provide returned data, but indication of of the vulnerability using out-of-scope tools.
- *Semi-blind* SSRF provides errors or partial server responses.
- *Non-blind* SSRF (*full-read* SSRF) provides a full response to the attacker indicating that the SSRF vulnerability exists.

## Preventing SSRF :

- Input validation ⇒ Allows rejecting dangerous characters passed to GraphQL arguments that accept URLs as part of a query or mutation.
- Network segmentation ⇒ Helps minimize the risk by ensuring that applications can communicate with only the relevant internal networks.
- Threat modeling ⇒ Allows identifying potential risks earlier in the development life cycle.
- Least privileges principle ⇒ Helps minimize the blast radius.

# Cross-Site WebSocket Hijacking (CSWSH)

- CSWSH is a CSRF vulnerability that impacts the handshake part of WebSocket communications, which use cookie-based authentication.
- It allows attacker to forges WebSocket messages on behalf of the victim and uses their authenticated session.

## Finding Subscription Operations :

- To get subscription-related fields, use this query :
    
    ```graphql
    query {
      __schema {
        subscriptionType {
          fields {
            name
          }
        }
      }
    }
    ```
    

## Hijacking a Subscription Query :

- How it happens :
    1. As attacker, open a Netcat TCP listener on port 4444 (wsl). 
        
        ```bash
        nc -vlp 4444
        ```
        
    2. As victim, we’ll simulate a user falling victim to a social-engineering attack by dropping an HTML file into the browser so it loads the JavaScript code, hijacking the user’s session to perform a WebSocket handshake and subscribe to the events.
        - Example of payload : https://github.com/dolevf/Black-Hat-GraphQL/blob/master/ch09/websockets_hijack.html
    3. Create a new paste.

## Preventing CSWSH :

- Because CSWSH is a CSRF attack, you can prevent it by using CSRF mitigation techniques.
- Validation of the Origin header is also crucial to preventing CSWSH attacks.
