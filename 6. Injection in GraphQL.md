# Injection Vulnerabilities in GraphQL

- Injection vulnerabilities occur when an application accepts and processes untrustworthy input without any sanitization.
- This allow the input to be interpreted as a command or a query and execute on either the client side or server side.

---

---

# The Injection Surface

- Consider these question while testing each input :
    - Does the application verify the incoming client input at all?
    - Does the application accept dangerous characters?
    - Does the application throw exceptions to unexpected characters?
    - Does GraphQL check the value type passed to the arguments?
    - Can we infer from the GraphQL server response (or out-of-band response) whether an injection attempt was successful?

## Query Arguments :

- GraphQL operations such as queries, mutations, and subscriptions can be designed to accept arguments like `limit` argument in the following query (read-only) :
    
    ```graphql
    query {
    	pastes(limit: 100) {
    		id
    		ipAddr
    	}
    }
    ```
    
    - `limit` value doesn’t allow us to execute code, but it can be used to make unexpected behaviors like returning all data when the value is `-1` .
    - `SELECT id, ipAddr FROM pastes LIMIT 100;`
- If we have a mutation (accept string values) like :
    
    ```graphql
    mutation {
    	createPaste(content: "Some content", title:"Some title", public: false) {
    		paste {
    			id
    			ipAddr
    		}
    	}
    }
    ```
    
    - It will result an SQL query like `INSERT INTO pastes (content, title, public) VALUES ('some_malicious_content', 'some_title', false)` .
- So, if you tried to exploit it with mutation like :
    
    ```graphql
    mutation {
    	createPaste(content: "content'); DELETE FROM users; --") {
    		paste {
    			id
    			ipAddr
    		}
    	}
    }
    ```
    
    - It will result a malicious query like `INSERT INTO pastes (content) VALUES ('content'); DELETE FROM users; --`

---

## Field Arguments :

- Field arguments aren’t different from other arguments, so it can be exploitable (when inserted into a database or influence logic) if it is in query like :
    
    ```graphql
    query {
    	users {
    	username(**capitalize: true)**
    		id
    	}
    }
    ```
    
- `SELECT id, CONCAT(UCASE(LEFT(username, 1)), LCASE(SUBSTRING(username, 2))) AS username
FROM users;`

---

## Query Directive Arguments :

- The query directives attached to certain GraphQL fields can also accept arguments like :
    
    ```graphql
    query {
    	pastes {
    		id
    		ipAddr @show_network(style: "cidr")
    	}
    }
    ```
    
    - `style` argument accept string value like “cidr” to make IP address in CIDR format.
- To get only the available directives :
    
    ```graphql
    query GetDirectives {
    	__schema {
    		directives {
    			name
    			description
    			locations
    		}
    	}
    }
    ```
    

---

## Operation Names :

- *Operation names* are strings we can add to GraphQL operations such as queries, mutations, or subscriptions.
- They are often used to uniquely name each query when multiple queries are sent together.
- For example :
    
    ```graphql
    query only_id{
    	pastes{
    		id
    	}
    }
    
    query only_title{
    	pastes{
    		title
    	}
    }
    ```

# SQL Injection

- SQLi vulnerabilities happen when client input is directly inserted into a SQL command without proper character escaping.

## Understanding the Types of SQL Injection :

### Classic SQL Injection :-

- *Classic SQLi* when the application returns SQL query errors displayed directly in the web page or become apparent through network inspection and happens with 2 techniques :
    - *Error-based SQLi* happens when the app throw SQL errors that help attacker to find the right pattern of exploit.
    - *Union-based SQLi* happens by leveraging the UNION SQL operator which concatenates results of multiple SELECT statements and return it to the client.

### Blind SQL Injection :-

- In *blind SQLi*, we are given no visible indications that a vulnerability exists while the app could fail silently and happens with 2 techniques :
    - *Time-based SQLi*  happens by supplying a SQLi payload that instructs the database to wait for a certain number of seconds, we can infer that the application is vulnerable if the delay occurs.
    - *Boolean-based SQLi* happens by constructing a payload that will return a Boolean result such as true or false.

---

## Testing for SQLi :

- Testing for SQLi can be done in a variety of ways, such as the following:
    - Submitting characters like single (') or double (") quotes and observing how the application handles unexpected inputs and errors.
    - Fuzzing input fields and observing application errors.
    - Submitting SQL commands that introduce delays.

    - It may makes SQL query like : `SELECT id, content, title FROM pastes WHERE content LIKE 'My First Paste'` .

## Automating SQL Injection :

- To automate this process, follow these steps :
    1. Put `*` in the filter place `pastes(filter:"test*")` .
    2. Copy the request and save it in *request.txt* file.
    3. Run this command to get successful payload : `sqlmap -r request.txt —dbms=sqlite —tables` .
    4. Run this command to list the tables : `sqlmap -r request.txt —dbms=sqlite -—tables` .

# Operating System Command Injection

- *Operating system (OS) command injection* vulnerabilities are injections that happen when user input is inserted into a system shell command.
- Web servers are often running under Unix accounts such as *www-data, apache, nginx*, or, if we get very lucky, the *root* user.
- When performing source code review, look for the following imported libraries and functions :
    - Python libraries such as *subprocess* and *os* and functions like `exec` and `eval` .
    - PHP functions such as `system`, `shell_exec`, `eval`, and `exec` .
    - Java functions such as `Runtime.exec()` .
    - Node.js modules such as `child_process` and functions like `exec` and `spawn` .

## An Example :

- Imagine we have a web server that accept URL from user and download it in its file system with flask code like :
    
    ```python
    @app.route('/download', methods=['POST'])
    def download():
     url = request.form['url']
     os.system('wget {} -P /data/downloads'.format(url))
    	return redirect('/dashboard')
    ```
    
- If the user puts this URL *http://example.com/file.zip,* it will be added in this system command directly : `wget -P /data/downloads http://example.com/file.zip` .
- Attack can exploit this by adding `;` to end command and inject its own command.

## Automated Testing with Commix :

- Some applications may restrict the types of characters they accept using firewalls.
- Command injections can happen by introducing any of the following characters, among others:
    - `;`  ⇒ To separate commands.
    - `&` ⇒ To send the first command to the background and continue to a second command we introduced.
    - `&&` ⇒ To run a second command after the first command finishes successfully (AND gate).
    - `||` ⇒ To run a second command after the first command finishes unsuccessfully (OR gate).
- To automate the process use this Commix command : `commix --url="http://127.0.0.1:5013/graphql" --data='{"query":"query{systemDebug(arg:\"test \")}"}' -p arg` .

---

## Code Review of a Resolver Function :

- Let’s perform a code review of the resolver function for `systemDebug` to see how it is implemented in DVGA. This should help us better understand the root cause of the OS command injection.

```python
def resolve_system_debug(self, info, arg=None):
	Audit.create_audit_entry(info)
	if arg:
		output = helpers.run_cmd('ps {}'.format(arg))
	else:
		output = helpers.run_cmd('ps')
	return output
```

- The vulnerability here is that there are no security checks on the supplied argument arg, so the resolver function will execute any Linux command it receives.
- This can be mitigated in multiple ways:
    - Accepting only alphabetic characters and ensuring that these are valid `ps` arguments.
    - Removing any dangerous characters.
    - Running the command as an unprivileged user to reduce the risk.
    - Using dedicated built-in libraries instead of shell commands directly, such as the `psutil` library in Python.

# Cross-Site Scripting

- It happen when client-side code (such as JavaScript) gets interpreted and executed within the context of a web browser.

## Reflected XSS :

- It occurs when input is submitted to the server and returned in an immediate response to the client.
- If you have this query to display a message :
    
    ```graphql
    query {
    	hello(msg:"Black Hat GraphQL")
    }
    ```
    
    - So, you can use this query to perform XSS by displaying document cookies :
        
        ```graphql
        query {
        	hello(msg:"<script>document.cookie;</script>")
        }
        ```
        

---

## Stored XSS :

- In *stored* XSS, the injection payload is persisted to a datastore rather than reflected to the client as part of a response to a query.
- The existence of the XSS payload in an application’s datastore could pose a risk to other systems, such as these:
    - Other servers reading the malicious input from the same datastore, so they are effectively impacted by the same exploit.
    - Other flows within the same GraphQL application reading from the same datastore.
- *Blind* XSS, a subcategory of stored XSS, happens when we inject the payload into a database table that’s read only by someone with a different level of access to the application.

---

## DOM-Based XSS :

- *Document Object Model–based XSS* vulnerabilities occur when a JavaScript injection payload gets executed exclusively within the browser’s DOM.
- The DOM is a representation of a web document that allows applications to modify their structure, content, and style.
- Because DOM XSS vulnerabilities happen in client-side code, GraphQL APIs aren’t the root cause of such vulnerabilities.
- Despite this fact, as community-built GraphQL clients could be vulnerable to these types of vulnerabilities.
- For a comprehensive list of the available GraphQL client libraries, visit https://graphql.org/community/tools-and-libraries/#javascript-client.
